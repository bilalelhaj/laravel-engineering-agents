# Walkthrough — Newsletter Subscription Feature

> **Note.** This walkthrough shows a **constructed minimal flow** (architect → builder → reviewer) — useful as a quick mental model. For a **real end-to-end run** on a Laravel 13 / Pest 4 project including parallel refinement (architect + db-architect + ui-ux), phase planning, and three full build/review cycles, see [`real-run.md`](real-run.md).

A complete end-to-end run of the three core agents on a small but realistic feature: *"a logged-in user can subscribe to a newsletter, and gets a confirmation email."*

This is a **constructed example** — the outputs below are representative of what each agent produces, not a transcript of one specific run. Run it on your own project to see the real outputs.

---

## 0. Setup

A fresh Laravel 12 project with Pest 3 and Mail/Notification facades available. No newsletter logic yet.

You drop the three agent files into `.claude/agents/` and start a Claude Code session at the project root.

## 1. Kicking off the pipeline

```
> Use @laravel-architect to plan a "logged-in user can subscribe to a
> newsletter and receive a confirmation email" feature, then
> @laravel-builder to implement, then @laravel-reviewer to audit.
```

The main conversation routes the first task to the architect.

---

## 2. `@laravel-architect` — clarifying questions

The architect inspects `composer.json`, `app/Models/`, existing migrations, then asks:

> **Q1.** Are subscriptions de-duplicated by email? (i.e. one user, one active subscription)
> **Q2.** Is the confirmation email *informational only*, or does it carry a confirmation link the user must click to activate?
> **Q3.** Should unsubscribed users be soft-deleted or hard-deleted?

You answer:
1. Yes, unique on `user_id`.
2. Informational — no confirmation link required for v1.
3. Hard delete; we don't need recovery.

The architect writes `docs/architecture.md`:

```markdown
# Architecture: Newsletter Subscription

_Generated by laravel-architect on 2026-05-01._

## Goal
Allow an authenticated user to create or remove a newsletter subscription
identified by their `user_id`. On creation, dispatch a confirmation email
asynchronously.

## Open questions
None.

## Database changes
- Migration: `2026_05_01_120000_create_newsletter_subscriptions_table.php`
- New table: `newsletter_subscriptions`
  - `id`, `user_id` (FK → users.id, cascadeOnDelete),
    `created_at`, `updated_at`
  - Unique index on `user_id`

## Eloquent
- New model: `App\Models\NewsletterSubscription`
  - `belongsTo(User::class)`
  - `casts()`: none beyond the timestamps default
- Modified: `App\Models\User`
  - `hasOne(NewsletterSubscription::class)` as `newsletterSubscription`

## Application layer
- Action: `App\Actions\Newsletter\SubscribeUser`
  - input: `User $user`
  - output: `NewsletterSubscription`
  - effect: idempotent — returns existing or creates; dispatches `NewsletterConfirmationMail`
- Action: `App\Actions\Newsletter\UnsubscribeUser`
  - input: `User $user`
  - output: `void`
- Routes:
  - `POST /newsletter` → `NewsletterController@store`
  - `DELETE /newsletter` → `NewsletterController@destroy`
  - both `auth:web` middleware
- No Form Request needed — no payload, the user is the only input
- API Resource: not needed (returns 201/204)

## Filament
Not in scope.

## Test strategy
- Feature: `tests/Feature/Newsletter/SubscribeTest.php`
  - it subscribes a logged-in user and dispatches the mail
  - it is idempotent on re-subscribe
  - it requires authentication (302 to login when guest)
- Feature: `tests/Feature/Newsletter/UnsubscribeTest.php`
  - it unsubscribes a subscribed user
  - it 204s when called by a never-subscribed user
- Fakes: `Mail::fake()`
- Factory: `NewsletterSubscriptionFactory` with one default state

## Out of scope
- Confirmation link / double opt-in
- Admin UI
- Rate limiting (separate concern, handled by the auth-throttle middleware)

## Implementation order
1. Migration + model + factory
2. SubscribeUser action + mailable + Feature test
3. UnsubscribeUser action + Feature test
4. Routes + controller + auth test
```

The architect returns to the orchestrator: *"Plan written to docs/architecture.md. Ready for the builder."*

## 4. `@laravel-reviewer` — audit

The reviewer reads only the diff, runs the suite, applies the checklist:

```markdown
# Review report — newsletter-subscription feature

_Generated by laravel-reviewer on 2026-05-01._

## Suite status
`./vendor/bin/pest` — PASS (9 tests, 23 assertions, 0 failures)

## Findings

### 🔴 Critical (0)
_None._

### 🟠 High (1)
- **H6 — Fat controller.** `app/Http/Controllers/NewsletterController.php:18-26`
  contains a `try/catch` that swallows the action exception and constructs a
  custom error response. Move error mapping to the exception handler in
  `bootstrap/app.php` so all controllers benefit, or rethrow.

### 🟡 Medium (2)
- **M5 — Group with `describe()`.** `tests/Feature/Newsletter/SubscribeTest.php`
  has 3 `it()` calls all using the same `$user = User::factory()->create()`
  setup. Wrap them in `describe('subscribing', fn () => ...)` with a
  `beforeEach` to remove the duplication.
- **M6 — Missing factory state.** `NewsletterSubscriptionFactory` has only the
  default. Add a `forUser(User $user)` state since two tests need it.

### 🔒 Security (0)
_None._

## Plan adherence
- [x] Migration with unique on `user_id`
- [x] `SubscribeUser` action — idempotent
- [x] `UnsubscribeUser` action
- [x] Mail dispatched (verified via `Mail::fake()` assertion)
- [x] Auth required on both routes
- [x] All planned Feature tests written
- [ ] Plan did not mention `Model::shouldBeStrict()` — builder added it as a
      cross-cutting improvement. Worth recording in the architecture doc.

## Recommended next steps
1. Address H6 in this PR.
2. Take 5 minutes for M5 + M6 — small wins.
3. Update `docs/architecture.md` with the `shouldBeStrict()` decision so the
   next architect run inherits the convention.
```

## 5. What this run shows

| | |
| :--- | :--- |
| **Round-trip from prompt to mergeable diff** | ~ 8–15 minutes on a real project, depending on suite size |
| **Each agent's context** | Stayed scoped — architect saw codebase, builder saw plan, reviewer saw diff |
| **Independent review caught real issues** | The fat controller and missing `describe()` were found *because* the reviewer didn't read the builder's transcript |
| **Plan adherence is automatic** | Reviewer cross-references the plan; "drift" is flagged not silently accepted |

## 6. When *not* to run the full pipeline

- A typo fix or one-line bug — direct prompt is faster
- An exploratory spike where you don't know the requirement yet — work in the main conversation, formalize once the shape is clear
- Pure UI/CSS tweaks with no business logic — the architect adds noise

The pipeline pays off when the change touches the database and needs tests.
